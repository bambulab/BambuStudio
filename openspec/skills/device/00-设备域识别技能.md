# 设备域识别技能

判别一个任务 / 文件改动 / commit / issue 是否属于"设备小组"的业务面。是所有其他设备域技能的前置判别器。

## 适用场景

- agent 接到一个新任务，但用户没有明示"进入设备域"
- 准备对 commit / push / review 应用设备域规则前
- 拉取上下文前，决定是否加载 `/openspec/` 设备 profile
- Code Review 时判断改动是否需要让设备小组介入
- 处理 Jira / Gerrit / GitHub PR 链接时，先识别归属再分流

## 不受 profile 开关限制（识别 ≠ 启用）

本技能**只输出识别结果**，是事实判定层，不是动作层：

- **不**读 `switch.json` 和 `STUDIO_AI_DEVICE_ENTRY` 作为前置门控，始终可用
- **不**自动加载 `/openspec/` 设备 profile
- **不**自动调用 `device-context-entry` / `device-knowledge-search` 等设备域其他技能
- **不**自动套用设备域规则（如 commit 标签 / Review 红线）

是否真正启用设备域上下文与技能，仍由原有三开关决定，本技能不替代它们：

1. `/openspec/ai-context/switch.json` 中 `deviceDevelopProfile == true`
2. 环境变量 `STUDIO_AI_DEVICE_ENTRY` 为 `true` / `1` / `on`
3. 用户在自然语言里明确要求进入设备域

识别为"设备域"只是给 agent 和用户一个**判断依据**——例如"这事看起来归你们小组管，要不要按设备域处理？"——决定权仍在用户和开关。

### 3. 弱信号（仅命中此 → 不足以判定，转步骤 4）

- 仅 `Plater.cpp` / `GUI_App.cpp` / `MainFrame.cpp` 等通用基建
- 仅 i18n / 翻译资源（`resources/i18n/*.po`）
- 仅打包 / 构建脚本（`CMakeLists.txt`、`build*.sh/.bat`）
- 仅命中 1 个关键词、且 path 无强信号

### 4. 显式信号 / profile 开关

显式信号同时影响"是不是设备域"和"是否启用 profile"：

- 用户消息含 "设备域 / 设备小组 / device profile / 进入设备域 / 按设备域处理" → 设备域，置信度 H，且视为已要求启用
- `/openspec/ai-context/switch.json` 中 `deviceDevelopProfile == true` → 已显式启用
- 环境变量 `STUDIO_AI_DEVICE_ENTRY` 为 `true` / `1` / `on` → 已显式启用
- 都未命中且前 3 步均无强 / 中信号 → **非设备域**（如不放心，向用户确认）

注意：**前 3 步只判定"是不是设备域"，不代表"已经启用 profile"**。即使强 / 中信号命中、识别为设备域，是否真正调用 `device-context-entry` 仍要看本步的三个开关——任一未启用，agent 应停在识别结论上、由用户决定下一步。

## 输出格式

判定结束后输出（结构化），并**不要直接调用其他设备域技能**——下一步由用户与开关决定：

```
设备域: 是 / 否 / 跨边界
置信度: H / M / L
触发信号:
- <信号 1>
- <信号 2>
profile 开关状态:
- switch.json deviceDevelopProfile: true / false
- STUDIO_AI_DEVICE_ENTRY: <值或 未设置>
- 用户显式要求: 是 / 否
建议下一步:
- 设备域 + 任一开关已启用 → 可按设备域流程继续（如 device-context-entry）
- 设备域 + 开关均未启用 → 提示用户"识别为设备域，是否启用 profile / 按设备域处理？"，等待用户决定，**不**自动加载 profile
- 跨边界 → 告知用户并询问归属与是否走双 review
- 非设备域 → 按通用上下文继续，不加载设备 profile
```

## 不做什么

- **不**因为识别为"设备域"就自动启用 profile / 调用 `device-context-entry` / 套用设备域规则——是否启用始终由 `deviceDevelopProfile` / `STUDIO_AI_DEVICE_ENTRY` / 用户明确要求决定
- 不在判定为非设备域时强行加载 device profile
- 不基于单一弱信号（仅 1 个 keyword / 仅 Plater 一处）就断言设备域
- 不把切片算法侧（FilamentGroup / MultiNozzleUtils / GCode 等）默认归为设备域，应走"跨边界"
- 不把"设备域识别"和"设备域 profile 进入"合二为一——前者是判别（本技能），后者是动作（由 `device-context-entry` 完成，且需开关或用户授权）
- 不把"机型 profile JSON"全盘归设备域，需要按改动字段细分
