# 文档沉淀技能

在完成设备相关开发后，把可复用知识补回 `openspec/` 下的设备目录。
所有设备域的归档物理上都落在 `openspec/`，不再写入 `.doc/`。

## 适用场景

- 完成功能开发后需要归档 OpenSpec change
- 发现了新的架构知识或代码入口需要记录
- 设备域文档有缺口需要补充

## 前置条件

- 了解 `openspec/rules/device/03-openspec规则.md` 的归档目标和命名规则
- 了解 `openspec/rules/device/01-目录与命名规则.md` 的目录职责

## 步骤

### 1. 判断归档位置

设备域归档统一落在 `openspec/` 下，按内容类型分流：

| 内容类型 | 归档到 |
| --- | --- |
| 功能分层、页面结构、数据流、设备能力关系 | `openspec/specs/device/功能架构文档/` |
| 源码入口、协议字段、关键类关系、跨切面参考（如主题与配色 / DPI 适配） | `openspec/specs/device/软件代码文档/` |
| 单个 OpenSpec change 的 proposal / design / tasks / specs 等完整产物 | `openspec/changes/device/<主题目录>/<YYYYMMDD改动内容>/` |
| change 沉淀完成后形成的稳定归档（按主题聚合的 README、基线规格、主题专题） | `openspec/specs/device/<主题目录>/` |

约定：

- 主题目录沿用现有命名（如 `耗材管理器/`），不另起新名
- `openspec/changes/device/<主题>/<YYYYMMDD改动内容>/` 是 change 进行中和落地后的「正本」
- `openspec/specs/device/<主题>/<改动名>/` 用作稳定视角的索引归档，内容应与对应 change 保持一致
- 不再把任何设备域内容写入 `.doc/`

### 2. 更新索引（每次归档必做）

按以下顺序逐一更新：

1. 对应主题目录的 `README.md`（如 `openspec/specs/device/耗材管理器/README.md`）
2. `openspec/specs/device/来源映射表.md` — 新增来源与归档位置映射
3. `openspec/specs/device/主题索引.md` — 在对应问题域下补充条目
4. `openspec/specs/device/软件代码文档/README.md` — 如变更记录数量变化则更新计数
5. `openspec/rules/device/03-openspec规则.md` — 如已纳入条目变化则更新

### 3. 最小沉淀检验

新增的每份文档至少能让后来者回答：

- 这份文档解决什么问题
- 它对应哪块设备能力
- 相关原始资料和源码入口在哪里

## 不做什么

- 不跳过索引更新直接放文件
- 不在 `openspec/specs/device/` 中创建与现有目录口径不一致的新目录
- 不把非设备域的通用资料放进 `openspec/specs/device/` 或 `openspec/changes/device/`
- 不再向 `.doc/` 写任何文件（该目录已废弃并删除）
