# 本地网络库查找技能

帮助开发人员在本地 `deps` / `dep_win_new` 历史中，按 slicer 版本前三段、编译档位、Internal/Public 和加固状态找到可用网络库 DLL。默认只输出到独立目录，不直接覆盖当前依赖。

## 适用场景

- 本地环境差异大，不确定应该从哪个 `deps` 目录拿网络库
- 需要区分 `Debug` / `Release`、`Internal` / `Public`、不同宏启用状态
- `deps` 仓库分支多、目录复杂，需要按版本、commit message 和 DLL 目录反查
- 需要拿非最新版本插件，或从 `deps` 历史里恢复旧网络库
- `deps` rebase 后，VS debug 因依赖变动要求全量编译才能启动
- 从 `dep_win_new` 拿到的插件存在 VMP-DLL 加固、版本不对等问题

## 强制流程

### 1. 从 slicer 推断需求，输出给用户并等待确认

先基于用户的 slicer 仓库推断本次要找的网络库条件。推断完成后，必须先向用户输出“需求摘要”，并等待用户确认或修正；**用户未确认前禁止进入 deps 查找**。

1. 读取 slicer 当前分支和 HEAD：
   - `git status --short --branch`
   - `git rev-parse --short HEAD`
2. 读取 Slicer 版本，优先级如下：
   - `version.inc` 中的 `SLIC3R_VERSION`
   - 已生成的 `libslic3r_version.h`（若存在）
   - 其它构建产物中的 `SLIC3R_VERSION`
3. 读取 `src/slic3r/Utils/bambu_networking.hpp`，仅作为历史兼容参考，不作为主要版本匹配来源：
   - `BAMBU_NETWORK_AGENT_VERSION`
4. 读取对应构建目录的 `CMakeCache.txt`：
   - `CMAKE_BUILD_TYPE`
   - `CMAKE_PREFIX_PATH`
   - Internal/Public 相关宏或选项
5. 读取已生成的 `libslic3r_version.h` 或 `version.inc` 中的 `BBL_RELEASE_TO_PUBLIC`：
   - `BBL_RELEASE_TO_PUBLIC=0` → **Internal**
   - `BBL_RELEASE_TO_PUBLIC=1` → **Public**
   - 未识别时必须在需求摘要中标为“未识别”，并让用户确认

需求摘要必须用表格展示：

| 项目 | 推断值 | 依据 |
| --- | --- | --- |
| slicer 仓库 | `<path>` | 当前 workspace / 用户输入 |
| slicer 分支 / HEAD | `<branch> / <hash>` | `git status` / `git rev-parse` |
| Slicer 版本 | `<xx.xx.xx.xx>` | `version.inc` / `SLIC3R_VERSION` |
| 需要的网络库版本 | `<xx.xx.xx.*>` | 取 Slicer 版本前三段 |
| 版本匹配规则 | `网络库与 Slicer 前三段匹配即可：xx.xx.xx.*` | Studio 与网络库兼容要求 |
| `BAMBU_NETWORK_AGENT_VERSION` | `<xx.xx.xx.xx>` | 仅作接口/历史兼容参考 |
| 构建档位 | `Debug / Release / 未识别` | `build_*` / `CMakeCache.txt` |
| Internal/Public | `Internal / Public / 未识别` | `BBL_RELEASE_TO_PUBLIC`：`0=Internal`，`1=Public` |
| 依赖根目录 | `<CMAKE_PREFIX_PATH>` | `CMakeCache.txt` |

输出需求摘要后，必须明确询问：

```text
请确认以上识别是否正确：Slicer 版本、需要的网络库前三段版本、Debug/Release、Internal/Public、是否需要未加固版本。
确认后我再进入 deps 查找；如有不对，请直接指出修正值。
```

### 2. 让用户提供 deps 仓库本地目录

输出需求摘要后，要求用户提供 deps 仓库根目录。即使从 `CMAKE_PREFIX_PATH` 推断出 `dep_win_new\usr\local`，也要确认它对应的 deps 仓库根目录。

### 3. 在 deps 库里创建查找 worktree

所有历史检索和恢复都在 deps 临时 worktree 中进行，避免污染原 deps 工作区。

```powershell
git -C "<deps>" worktree add "<deps>\.worktrees\network-library-search-<timestamp>" <start-branch-or-commit>
```

若 deps 原仓库不干净，不影响 worktree 查找；但必须在报告中说明原仓库存在未提交改动。

### 4. 先查与 slicer 匹配的网络库分支，找到就停止

先推断“与 slicer 匹配的网络库分支”，再查找。常见匹配线索：

- slicer 分支名中包含的版本号，例如 `release/v2.7.0_beta`
- Slicer 版本前三段，例如 Slicer `02.07.00.53` 对应网络库 `02.07.00.*`
- deps 分支名中的 `github` / `master` / release 标识
- commit message 中补充的网络库版本号

查找策略：

1. 在匹配分支查 commit message：
   - Slicer 前三段版本：例如 `02.07.00`
   - 精确 Slicer 版本可作为弱参考：例如 `02.07.00.53`
   - 关键词：`bambu_network` / `network module`
   - **默认只回溯近两个月历史**；除非用户明确要求更早版本，否则超过两个月的 deps commit 直接丢弃，避免无界搜索。
2. 对候选 commit checkout 到临时 worktree。
3. 按 deps 目录规则搜索 DLL。
4. 读取 DLL 版本、路径分类和加固状态，按匹配指标评分。
5. 如果找到满足条件的 DLL，停止后续分支查找，进入结果汇报。

### 4.1 无版本号 commit 的时间锚点策略

deps 的网络库 commit message 经常只有 `ci: update network module based on commit <network-sha> <timestamp>`，没有四段版本号。此时按以下方式缩小范围：

1. 在 slicer 仓库查 `src/slic3r/Utils/bambu_networking.hpp` 的近两个月历史，找 `BAMBU_NETWORK_AGENT_VERSION` 变更或 `ci: update network module...` 的时间点。
2. 以该时间点为锚，在 deps 仓库近两个月网络库 commit 中找前后相邻候选。
3. 只 checkout 这些候选 commit，读取真实 DLL 内部版本字符串。
4. 若 slicer 头文件当前 `BAMBU_NETWORK_AGENT_VERSION` 与 `SLIC3R_VERSION` 前三段不一致，仍以 `SLIC3R_VERSION` 前三段为主匹配；`BAMBU_NETWORK_AGENT_VERSION` 只作接口参考和提交时间锚点。

### 5. 匹配分支未找到时，再查 master，找到就停止

只有匹配分支未找到可用 DLL 时，才进入 `master` 分支。方法与匹配分支相同：查 commit message、checkout 候选 commit、搜索 DLL、读取版本、判断路径分类和加固状态。找到满足条件的 DLL 后停止。

## deps 网络库目录判定规则

按路径判断候选 DLL 的 `Debug/Release`、`Internal/Public`、加固状态和用途。

| 路径模式 | 档位 | Internal/Public | 加固状态 | 用途 / 备注 |
| --- | --- | --- | --- | --- |
| `usr/local/bin/bambu_networking.dll` | Release | Public | 加固 + 签名后 | 公开 Release 版本，线上交付常用 |
| `usr/local/bin/bambu_networking.dll.vmp` | Release | Public | VMPProtect 配置 | 加固工具配置，不是运行 DLL |
| `usr/local/bin/bambu_networking.dll.ssp` | Release | Public | Virbox 配置 | 加固工具配置，不是运行 DLL |
| `usr/local/bin/Debug/bambu_networking.dll` | Debug | Public | 通常未加固 | Debug 公开版，也走 release 路径体系 |
| `usr/local/bin/Debug/internal/bambu_networking.dll` | Debug | Internal | 通常未加固 | Debug + internal，常见于无 `-R` 产物 |
| `usr/local/bin/internal/bambu_networking.dll` | Release | Internal | 加固后 | internal Release 版本 |
| `usr/local/bin/internal/bambu_networking.dll.vmp` / `.ssp` | Release | Internal | 加固配置 | 工具配置，不是运行 DLL |
| `usr/local/bin/internal/not_protected/bambu_networking.dll` | Release | Internal | 未加固备份 | 加固前明文备份，适合排查导出/调试 |
| `usr/local/bin/internal/protected/bambu_networking.dll` | Release | Internal | 加固中间产物 | 加固工具中间产物 |
| `usr/local/bin/internal/win_arm64/bambu_networking.dll` | Release | Internal | ARM Windows 加固版 | ARM64，不适合 x64 Studio |
| `usr/local/bin/not_protected/bambu_networking.dll` | Release | Public | 未加固备份 | 公开 Release 加固前明文备份 |
| `usr/local/bin/protected/bambu_networking.dll` | Release | Public | 加固中间产物 | 加固工具中间产物 |
| `usr/local/bin/win_arm64/bambu_networking.dll` | Release | Public | ARM Windows 加固版 | ARM64，不适合 x64 Studio |

选择原则：

1. x64 Studio 默认排除 `win_arm64/`。
2. Debug 需求优先选 `Debug/` 下 DLL；Internal Debug 优先选 `Debug/internal/`。
3. Release Public 运行优先选 `usr/local/bin/bambu_networking.dll`；若需要排查导出或加固问题，再看 `not_protected/`。
4. Internal Release 优先选 `internal/bambu_networking.dll`；若怀疑 VMP/壳影响调试，再看 `internal/not_protected/`。
5. `.vmp` / `.ssp` 是加固配置，不作为候选 DLL。
6. Debug/Internal 未加固默认候选是 `usr/local/bin/Debug/internal/bambu_networking.dll` + `usr/local/lib/Debug/internal/bambu_networking.lib`；Release/Internal 未加固备份才看 `usr/local/bin/internal/not_protected/`。
7. 输出给开发者拷贝的文件名必须保持原名：`bambu_networking.dll` / `bambu_networking.lib`。不要改成 `Debug_Internal_...`；用目录层级区分用途。

## 版本匹配规则

Slicer 与网络库版本以四段号表示：`xx.xx.xx.xx`。**匹配主依据是 Slicer 版本前三段**，不是 `BAMBU_NETWORK_AGENT_VERSION`。

- **推荐匹配**：网络库前三段与 Slicer 版本前三段一致，例如 Slicer `02.07.00.53` 匹配网络库 `02.07.00.*`
- **强参考**：四段完全一致时可加分，但不是必要条件
- **不匹配**：前三段不同，例如 Slicer `02.07.00.53` 与网络库 `02.06.01.50`

结果中必须同时展示“Slicer 前三段匹配”和“四段是否一致/版本字符串”两个指标，不要只写“版本一致/不一致”。

## 候选 DLL 匹配指标

对每个候选网络库输出以下指标，并给出“最匹配列表”。

| 指标 | 匹配要求 |
| --- | --- |
| Slicer 前三段版本 | DLL 字符串 / 文件版本 / commit message 是否与 Slicer 版本前三段一致；这是主要兼容要求 |
| 四段版本 | DLL 字符串 / 文件版本 / commit message 是否与 Slicer 四段完全一致；仅加分参考 |
| 分支 | 优先匹配 slicer 对应网络库分支，其次 master |
| 构建档位 | Debug / Release 与 slicer 构建档位一致 |
| Internal/Public | 与 slicer 宏或用户要求一致 |
| 加固状态 | Debug 场景优先未加固；Release 按 Public/Internal 交付要求处理 |
| 架构 | x64 Studio 排除 `win_arm64` |
| commit | deps commit 与 slicer 需求时间线接近，且 commit message 可信 |

## 结果汇报格式

查找结果必须用 Markdown 表格展示，至少包含：

| 排名 | DLL | 来源分支 | 来源 commit | 档位 | Internal/Public | 网络库版本 | Slicer 前三段匹配 | 四段一致 | 加固状态 | 架构 | 评分 | 结论 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | `<path>` | `<branch>` | `<hash subject>` | `Release` | `Public` | `02.07.00.xx` | `是` | `<是/否>` | `加固/未加固` | `x64` | `<score>` | `推荐` |

输出目录也用表格展示：

| 内容 | 路径 / 值 |
| --- | --- |
| 输出目录 | `<path>` |
| 推荐 Release DLL | `<path>` |
| 推荐 Debug DLL | `<path>` |
| 推荐 Debug LIB | `<path>` |
| 拷贝建议 | `请开发人员手动拷贝到目标 runtime / install 目录` |

输出目录建议保留原文件名，并用目录表达条件：

```text
<output>/
  Debug/internal/bambu_networking.dll
  Debug/internal/bambu_networking.lib
  Release/internal/bambu_networking.dll
  Release/internal/bambu_networking.lib
```

## 不做什么

- 不在没有用户确认时修改 deps 原仓库分支历史
- 不直接把 DLL 复制进 slicer install 目录
- 不把 `.vmp` / `.ssp` 当作候选 DLL
- 不把 `win_arm64` DLL 推荐给 x64 Studio
- 不只看文件名判断版本，必须核验版本字符串 / 文件版本 / commit 来源
