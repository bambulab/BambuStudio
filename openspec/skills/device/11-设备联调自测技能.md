# 设备联调自测技能

把一次改动的 spec 文档里"应该变成什么样"（改动预期）和"用户能做什么"（功能预期）沉淀为一份
可交互的本地网页，放回改动自己的 spec 目录。开发者在联调时就着这份清单逐条按
**通过 / 失败 / 跳过** 三态勾选，并写备注（支持 `Ctrl+V` 粘贴截图），页面会自动保存到
`localStorage`，结束后可一键**复制 Markdown** 报告或**导出 JSON** 进度快照。

本技能**只负责生成页面与（可选的）整理报告**，不替开发者做测试，也不触发构建、调试或提交。

## 触发条件

本技能**不跟随 `deviceDevelopProfile` 开关自动激活**。

仅以下两种情况才允许触发：

1. 用户在 Cursor UI 中主动选中 / 调用 `device-self-test`；
2. 用户在自然语言里明确说"生成联调自测页""把 spec 转自测清单""按设备联调自测技能处理"
   "按 `device-self-test` 生成一份自测 HTML"等等价表达。

下列情况**不**视为触发：

- 用户说"帮我跑一下自测""我要测一下"等一般性表述；
- 上游技能（`device-dev-completion`、`device-gerrit-push` 等）内部觉得应该测；
- AI 自行判断"该出一份自测清单"。

不确定是否被触发时，默认不执行，先追问用户。

## 适用阶段

- 推荐插入在 **阶段 05（本地构建调试）完成之后、阶段 07（本地提交）之前**；
- 也可以在阶段 08（推送前自查）跑一遍作为旁证；
- 本技能与 `device-local-build` / `device-local-debug` 互补：那两个是把东西跑起来，本技能是
  把"该验证哪些东西"显式化。

## 输入

- **必填**：一个改动的 spec 目录路径，例如：
  `openspec/specs/device/软件代码文档/耗材管理器/20260417耗材管理器云端接入Web前端/`
- **可选**：改动标题（不给则从 `proposal.md` H1 抽取）
- **可选**：Jira / Gerrit 链接（不给则从 proposal / 最新 commit 里找）
- **可选**：补充条目（用户临时想加的自测点）

## 执行步骤

### 1. 门控 + 告知

严格按"触发条件"。触发后先用 Markdown Alert 告知：

```md
> [!IMPORTANT]
> 已进入 `device-self-test`（设备联调自测技能）。
> - spec 目录：<绝对 / 相对路径>
> - 将扫描：proposal.md / design.md / tasks.md / specs/**/spec.md（存在即读）
> - 产出：<spec-dir>/self-test.html（可能覆盖同名文件）
> - 不做：构建、调试、commit、推送 Gerrit、回写 Jira
> 抽取完成后会先列给你看，确认后再写文件。
```

### 2. 定位 spec 目录

按顺序尝试：

1. 用户直接给出 spec 路径 → 直接用；
2. 用户只给改动标题 / Jira → 在 `openspec/specs/device/软件代码文档/**/` 下搜 proposal.md
   包含该标题或 Jira 的目录；
3. 用户没给 → 从当前分支名或 `git log -1` 推断；
4. 以上都失败 → 向用户追问，**不**自己猜测创建新目录。

### 3. 抽取预期

读取以下文件（存在即读，不存在则跳过并在产出页提示）：

| 文件 | 抽取内容 | 默认 group.type |
| --- | --- | --- |
| `proposal.md` | 改动目标 / 价值 / 范围 / 非范围 | `change` |
| `design.md` | 架构决策 / 接口契约 / 兼容性 / 风险 | `change` |
| `tasks.md` | 每个任务的"完成标志 / 验收标准" | `feature` |
| `specs/**/spec.md` | Requirements / Acceptance Criteria / User Story | `feature` |

**分组组织方式**（关键）：分组 = **一个可独立验证的业务维度**（视图 / 功能模块 / 场景），
不是简单把 change / feature 各塞一个组。参考耗材管理器实例：
`F1 页面入口与导航 / F2 列表浏览与筛选 / F3 详情 / F4 添加 / ... / F8 云端同步 / F9 调试日志`。

- 每个分组最好 ≤ 10 个条目；超了就拆；
- 分组 id 用有语义的短前缀：`C1 / C2 ...` 归 `type: "change"`，`F1 / F2 ...` 归 `type: "feature"`；
  也允许混合（如 `AMS1 / UI1`），只要 id 在全页内唯一；
- 每个条目是一次可执行、可观察的操作 → 预期对。

每条抽成：

```json
{
  "id": "F1.1",
  "op": "启动 Studio，点击 MainFrame 顶级 Tab <b>耗材管理</b>",
  "exp": "进入 <code>FilamentPage</code>，默认视图为\"我的耗材\""
}
```

抽取原则：

- **op（操作）**：写成具体动作——"点 X""输入 Y""连接设备后 Z"；避免抽象的"测试 W"；
- **exp（预期）**：写成可判真假的观察结果——"出现 toast""日志有 POST""列表条数 +1"；
  避免主观感受"显示正常""性能 OK"；
- `op` / `exp` 可内嵌少量 HTML（`<code>`、`<b>`、`<i>`）增强可读性；
- 一条 op + exp 控制在 2 行之内；更长就拆；
- **禁止**把纯内部实现（"调用 `FilaCloudApi::onLogin`"）当作 exp；应翻成用户 / 日志 / HTTP 可见的现象；
- 若 `tasks.md` 把改动点和功能点混在一起，按**用户能否感知**拆：
  用户能感知 → `type: "feature"`；仅工程师 / 内部指标感知 → `type: "change"`。

### 4. 向用户确认

抽取完以紧凑列表形式给用户过目，等**一次用户确认**再写文件：

```md
> 拟生成 3 个分组 / 18 条自测点：
> C1 · 协议新字段（change，4 条）：C1.1 … C1.4
> F1 · 页面入口与导航（feature，5 条）：F1.1 … F1.5
> F2 · 云端同步（feature，9 条）：F2.1 … F2.9
> 回复「ok」写文件，或指出要加 / 改 / 删的分组或条目。
```

### 5. 生成 HTML

1. 读模板：`openspec/skills/device/templates/self-test-template.html`
2. 定位其中 `<script id="self-test-data" type="application/json"> ... </script>` 内部的 JSON；
3. **整段替换** JSON 内容（保留外层 `<script>` 标签），字段参考下面 schema；
4. 不要改模板的 `<title>` 或 `<h1>` 文本——它们在运行时由 `SELF_TEST_DATA.title` 覆盖；
5. 用 `Write` 工具写到 `<spec-dir>/self-test.html`。

JSON 字段：

```json
{
  "title": "改动标题（自测页标题 + h1 + 浏览器标签）",
  "spec_path": "openspec/specs/device/软件代码文档/<模块>/<日期+主题>",
  "storage_key": "<slug>-selftest/v1",
  "export_prefix": "<slug>-selftest",
  "generated_at": "ISO8601",
  "jira": "STUDIO-xxxxx 或空",
  "gerrit": "change URL 或空",
  "groups": [
    {
      "id": "F1",
      "type": "feature",
      "name": "页面入口与导航",
      "summary": "Tab · 左导三视图 · 主题跟随 · 退出保活",
      "items": [
        {
          "id": "F1.1",
          "op": "启动 Studio，点击 MainFrame 顶级 Tab <b>耗材管理</b>",
          "exp": "进入 <code>FilamentPage</code>，默认视图为\"我的耗材\""
        }
      ]
    }
  ]
}
```

`storage_key` / `export_prefix` 建议规则：

- 用改动目录名的 kebab-case slug，例如
  `20260417耗材管理器云端接入Web前端` → `fila-cloud-web-selftest`；
- `storage_key` 追加 `/v1` 便于将来数据结构升级；
- 两者可相同前缀，例如 `storage_key: "fila-cloud-web-selftest/v1"`、
  `export_prefix: "fila-cloud-web-selftest"`。

### 6. 给出打开方式

写完后提示用户：

```md
已生成：`<spec-dir>/self-test.html`（X 个分组 / 共 N 项自测点）
打开方式（PowerShell）：
```powershell
start "<spec-dir>\self-test.html"
```
- 状态三态：✓ 通过 / ✕ 失败 / — 跳过（再点 ↺ 清空）
- 备注 textarea 支持 Ctrl+V 粘贴截图；截图会压缩后存进 localStorage
- 页面进度按 `storage_key` 本地保存；换浏览器 / 清缓存会丢
- 想留存：点「导出」下 JSON，或点「复制MD」把当前报告复制到剪贴板
```

### 7. 收尾（可选）

用户反馈"测完了"时，可协助：

- 让用户点页面顶栏「复制MD」→ 把剪贴板内容粘到
  `<spec-dir>/self-test-report-YYYY-MM-DD.md` 入库；
- 或让用户点「导出」下 JSON 作为进度快照存档；
- 若要把报告回写 Jira / Gerrit，转交 `device-jira-comment` / `device-gerrit-ops`；
- **不要**替用户在网页里填状态 —— 状态由开发者亲自勾。

## 不做什么

- ❌ 不触发 `device-local-build` / `device-local-debug`；
- ❌ 不跑任何测试命令；
- ❌ 不替开发者勾选状态、编造通过结论；
- ❌ 不提交 HTML 或报告到 git（由开发者决定是否入库）；
- ❌ 不回写 Jira / Gerrit；
- ❌ 不修改 spec 源文件（proposal / design / tasks / spec）；
- ❌ 不在 HTML 里嵌入凭据、token、绝对本机路径；
- ❌ 不下载外部资源，模板必须完全自包含。

## 产出

- `<spec-dir>/self-test.html` 或 `<spec-dir>/自测/index.html`（可交互自测页，单文件 HTML）
- 可选：`<spec-dir>/self-test-report-YYYY-MM-DD.md`（联调结束后"复制 Markdown"粘贴入库）
- 可选：`<spec-dir>/self-test-progress-YYYY-MM-DD.json`（进度快照，可导入他人的页面续测）

## 模板

共享模板：`openspec/skills/device/templates/self-test-template.html`

- 无外部依赖（纯 HTML + CSS + 原生 JS），自动适配明暗主题；
- 自动保存到 `localStorage`，key 由注入的 `storage_key` 字段决定；
- **三态**：`pass` / `fail` / `skip`（+ `reset` 清除）；
- **顶部 sticky 进度条**：三色分段（绿 pass / 红 fail / 灰 skip），
  文字形如 `12 pass · 1 fail · 2 skip · 3 pending / 18`；
- **左侧 sticky TOC**：各分组完成进度 + 有失败条目时高亮；
- **表格列**：`ID | 操作 | 预期 | 状态按钮组 | 备注`；
- **备注支持 `Ctrl+V` 粘贴截图**：自动压到长边 1280 / JPEG 0.82，
  缩略图 + lightbox 放大，× 号删除；
- **工具栏**：展开 / 折叠全部 / 导出 JSON / 导入 JSON / 复制 Markdown / 重置；
- **导出 JSON** 包含 `title / spec_path / storage_key / exported_at / state{pass fail skip note images}`，
  便于他人导入复盘；
- **复制 Markdown** 会把当前所有状态和备注整理成结构化报告到剪贴板，可直接粘到 Jira / Gerrit / 文档；
- 窄屏（< 960 px）：TOC 自动回到顶部、隐藏备注列，表格可横滚。

实例参考：`openspec/specs/device/软件代码文档/耗材管理器/自测/index.html`。

## 异常处理

| 现象 | 处理 |
| --- | --- |
| spec 目录不存在 | 不自己创建；让用户确认路径或重新给 |
| proposal / design / tasks 都缺 | 只用 specs/**/spec.md；若也没有，先让用户手动提供 items |
| 已存在 `self-test.html` | 默认覆盖，但先告知用户"将覆盖 N 字节的旧文件"，等确认 |
| 用户给的是非 spec 目录（例如源码目录） | 拒绝写入，要求指向 `openspec/specs/device/...` 下 |
| 抽出来只有 0-2 项 | 提示用户补充，或让用户口述关键自测点后再生成 |

## 与其他技能的关系

| 技能 | 关系 |
| --- | --- |
| `device-knowledge-search` | 找 spec 目录时可以复用 |
| `device-problem-analysis` | 问题单没有 spec 时，先用它产出轻量 spec，再来这里生成自测页 |
| `device-local-build` / `device-local-debug` | 并列的本地工具；本技能只管"要测什么"，它们管"怎么跑起来" |
| `device-change-review` | 推送前自查：可把本自测页的通过率 / 失败项作为一个输入 |
| `device-jira-comment` | 把导出的 Markdown 报告作为"测试结论"回写 Jira |
| `device-doc-archiving` | 如果自测中发现新知识点，归档走它 |
