# 设备开发完成技能

开发周期结束后执行善后流程，确保知识沉淀和能力积累。

## 适用场景

- 功能模块开发完成并编译通过
- bug 修复或功能增强结束
- OpenSpec change 实现完毕，准备归档

## 善后清单

按顺序执行 5 个阶段。

### 阶段 1：确认交付物

- 代码编译通过，无新增错误
- OpenSpec artifact 状态为 `done`（如有）
- 前端 dev server 能正常启动（如有前端改动）
- 待验证项已记录（不阻塞善后）

### 阶段 2：文档沉淀

按 `04-文档沉淀技能.md` 执行：

1. 判断归档类型（OpenSpec / 架构知识 / 代码入口）
2. 归档到 `openspec/specs/device/` 对应位置
3. 更新全部索引文件
4. 最小沉淀检验

### 阶段 3：本地提交

按 `05-设备代码本地提交技能.md` 执行：

1. 盘点改动 → 拆分 commit → 提交 → 验证

### 阶段 4：流程回顾

回顾四个维度，输出改进清单：

| 维度 | 关键问题 |
| --- | --- |
| 文档缺口 | 开发中是否因缺文档多花时间？索引是否覆盖本次知识点？ |
| 规则缺口 | 是否遇到规则未覆盖的决策？现有规则是否过时？ |
| 技能缺口 | 是否有重复操作可抽象为新技能？现有技能是否有遗漏？ |
| 流程摩擦 | 哪个步骤耗时最多？上下文加载是否顺畅？ |

### 阶段 5：落地改进

| 改进类型 | 处置 |
| --- | --- |
| 规则/技能小幅修订 | 直接编辑对应文件 |
| 新增规则/技能 | 创建文件并更新 README |
| 文档补充（10 分钟内） | 立即补 |
| 大改动 | 记录追踪，不阻塞收尾 |

落地后按 `05-设备代码本地提交技能.md` 提交。

## 最小执行路径

| 模式 | 阶段 |
| --- | --- |
| 保底 | 1 + 3 |
| 标准 | 1 + 2 + 3 |
| 完整 | 1 + 2 + 3 + 4 + 5 |

## 与其他技能的关系

```
本技能编排:
  阶段 2 → 04-文档沉淀技能
  阶段 3 → 05-设备代码本地提交技能
  阶段 5 → 05-设备代码本地提交技能
后续可选:
  06-设备改动Review技能 → 07-设备代码推送Gerrit技能
```

## 不做什么

- 不跳过阶段 1 直接归档或提交
- 大改动不强行在收尾阶段完成
- 不替代 Gerrit Code Review
