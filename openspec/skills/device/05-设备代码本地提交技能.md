# 设备代码本地提交技能

在设备域开发完成后，把改动按逻辑分组提交到本地仓库。

## 适用场景

- 完成设备域功能开发、bug 修复或功能增强后提交
- 完成设备文档归档或规则更新后提交
- 一次开发周期中同时产出了代码和文档

## 前置条件

- C++ 代码已编译通过（如有代码改动）
- 文档归档已按 `03-openspec规则.md` 完成索引更新（如有归档）

## 步骤

### 1. 盘点改动

```powershell
git status
git diff --stat
```

按以下类别分组：

| 类别 | 典型文件 |
| --- | --- |
| 功能代码 | `src/slic3r/`, `CMakeLists.txt` |
| OpenSpec 源文件 | `openspec/changes/`, `openspec/specs/` |
| 设备域文档 | `openspec/specs/device/`, `openspec/rules/device/` |
| 无关文件（不提交） | `.specify/`, `.superpowers/`, `.vscode/`, `dev_build.bat` |

### 2. 拆分 commit

一个逻辑变更对应一个 commit：

- **纯文档更新** → 独立 commit
- **功能代码 + OpenSpec + 归档 + 索引** → 一个 commit
- 关联性强且不可分割 → 合并为一个

### 3. 构造 commit message

格式参见 `/.cursor/rules/commit-msg.mdc`：

```
[Change Type]: 简短描述

（选填）详细说明

jira: STUDIO-数字 | github pull request: PR链接 | jira: none reason: <reason>
自测结果: pass | fail（fail 需备注）
```

**AI 编写时的字数限制（必守）**：

- 标题 `[Change Type]: 简短描述`：单行 ≤ 12 词（≤ 60 字符），不写句号
- 正文（详细说明）：**总长度 ≤ 30 词**（约 ≤ 60 个汉字），只保留 1 个自然段
  - 只写「做了什么 / 为什么这么做」，不复述 diff、不罗列文件
  - 行为变化、风险点、非直观实现优先；语义可推断的细节一律省略
  - 超过 30 词必须删减，不要换成「分多段」绕过限制
- trailer（jira / 自测结果 / Change-Id）不计入正文字数

**常用 Change Type**：

| 场景 | Type | reason |
| --- | --- | --- |
| 新功能模块 | NEW | other |
| 增强已有功能 | ENH | other |
| 修复缺陷 | FIX | — |
| 仅文档/规则 | ENH / ADD | doc-only |

### 4. 暂存与提交

PowerShell 用 `@""@` 传递多行 message：

```powershell
git add <文件列表>

$msg = @"
NEW: 简短描述

详细说明

jira: none reason: other（备注）
自测结果: pass
"@

git commit -m $msg
```

- 中文路径用双引号包裹
- 提交前 `git diff --cached --stat` 确认暂存内容

### 5. 验证

```powershell
git log --oneline -5
git status
```

确认 commit 出现在历史中、无遗漏文件、不相关文件仍 untracked。

## 不做什么

- 不提交无关文件（`.specify/`、`.superpowers/`、IDE 配置等）
- 不提交含密钥的文件（`.env`、`credentials.json`）
- 不 `git push`，除非用户明确要求
- 不修改 git config
- 不使用 `--force` / `--amend`，除非用户明确要求
