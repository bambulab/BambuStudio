# 设备上下文接入规则

本规则用于定义设备研发目录如何与仓库中的 AI 上下文开关、设备 profile 和规则入口衔接。

目标是明确两件事：

- 什么时候应启用设备研发上下文
- 启用后应如何进入 `openspec/` 下的设备目录

## 1. 开关来源

设备 profile 由两条等价开关 + 用户显式要求三选一启用，任一成立即视为打开：

1. `/openspec/ai-context/switch.json` 中：

   ```json
   {
     "deviceDevelopProfile": true
   }
   ```

2. 环境变量 `STUDIO_AI_DEVICE_ENTRY` 设置为 `true` / `1` / `on`（不区分大小写）。
   适用于 CI / 临时会话 / 不能写 `switch.json` 的场景。
3. 用户在自然语言里明确要求进入设备域（例如「按设备域处理」「使用设备域 XX 技能」）。

## 2. 未启用时的规则

如果上述三路开关都未触发，则：

- 不主动把 `/openspec/docs/device/` 作为默认上下文
- 不强行套用设备研发规则
- 非设备组任务继续按仓库通用上下文处理

## 3. 启用后的统一入口

当任一开关条件成立且当前任务属于设备域时，优先阅读：

1. `/openspec/ai-context/profiles/device/entry.md`
2. `/openspec/ai-context/profiles/device/load-order.md`

然后再回到设备研发目录正文：

1. `/openspec/docs/device/readme食用指南.md`
2. `/openspec/docs/device/开发流程.md`
3. 若任务涉及设备域 `OpenSpec`，先读 `/openspec/rules/device/03-openspec规则.md`
4. `/openspec/specs/device/README.md`
5. 需要快速定位问题域时，再看 `/openspec/specs/device/主题索引.md`

## 4. 执行原则

- 不要全量扫描 `/openspec/docs/device/`
- 先走入口，再走索引，再按相关性深入
- 设备域任务的回答、实现、分析和文档整理应优先与 `/openspec/docs/device/` 中已有约束保持一致

## 5. 归档边界

以下内容属于设备域接入规则，应归档到 `openspec/rules/device/` 或 `openspec/skills/device/`：

- 设备 profile 的启用条件
- 设备 profile 的加载顺序
- 设备域 `OpenSpec` 的优先入口要求
- 设备主题索引与来源映射的入口链路

以下内容默认不归档到 `openspec/` 下的设备目录：

- 通用输出格式规则
- 与设备域无关的通用 agent 表达规范
- 不依赖设备知识目录的通用提示词资产
