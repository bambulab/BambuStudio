## 2. 跨多单的横切要点（必看）

下面这些「现象不同、底层一回事」的模式值得在评审 / 设计阶段就想到。

### 2.1 macOS WKWebView 不要做同步 JS 调用

| 反例 | 正例 |
| --- | --- |
| 同步 `evaluateJavaScript` + 立刻读取返回值 | async `evaluateJavaScript` + completion handler |
| Tab show 时同步 NavigateTo + SetFocus | `CallAfter(NavigateTo)`，跳过对 webview 的 SetFocus |

WKWebView 在 macOS 上对同步 JS 调用相当敏感，主线程一阻塞就表现成「点了 Tab 没反应」。
Win / Linux 路径基本不受影响，但写代码时按 macOS 标准来，避免分支化。

### 2.2 原生 `<input type="color">` 的 onChange 不能直接绑 form

`<input type="color">` 的 `onChange` 在用户拖动 hue 滑块时会持续触发，每次都覆盖一次 form 状态。
正确姿势：

- 维护一个本地 `draft` 状态
- 只在 OK / Confirm 这一类显式确认动作里调用统一入口（这里是 `commitCustomColorSelection`），让它做 dedupe / preset 重叠校验
- Cancel、Esc、click-outside 一律丢弃 draft，保留旧值

复用此模式时务必走 `commitCustomColorSelection` 这一个口，不要在多个地方各写一份提交逻辑。

### 2.3 编辑类云端 PUT payload 必须把"显示字段"也传回去

`UpdateFilamentV2` 这种"全量替换式"的 PUT，云端拿不到 `filamentName` 会直接 400。
教训：

- payload 构造函数在编辑分支不要按字段做差量
- 至少把云端做主键、做去重、做展示用的核心字段（`filamentId` / `filamentName` / `setting_id`）带全
- 写完调一次 round-trip 测试（创建 → 拉取 → 编辑非名字字段 → 拉取）

### 2.4 切片兼容过滤 vs UI 选择过滤要分清

下拉框分两类：

| 类型 | 是否参与切片兼容 | 是否按机型过滤 |
| --- | --- | --- |
| 切片参数预设（slicing preset） | 是 | 是 |
| 品牌 / 耗材类型 / vendor 这类 UI metadata | 否 | 否 |

`build_preset_options()` 错就错在把第二类按第一类过滤，导致 macOS 上品牌列表残缺。
新增"按机型过滤"逻辑前先问一句：这个字段会进切片器吗？不进就不要过滤。

### 2.5 列表筛选要与权威面板对齐

耗材管理器的材料列表必须和 `AMSMaterialsSetting::Popup` 保持同一过滤口径：

- `compatible_printers`
- root preset gate
- user-preset gate
- `filament_id` 去重

任何一处漂移都会导致同一台机器在两个入口看到不同的耗材集合，用户即时炸。

### 2.6 前端本地匹配优先 ID、再 fallback 名字

云端给前端的 spool 数据带 `filamentId`，本地预设也带（或可以反查）。
本地匹配顺序固定：

1. `filamentId` 完全相等
2. 才落到 `filamentName` 字符串比对

依赖 `filamentName` 容易因为 vendor 前缀、locale、空格之类的差异错配。

### 2.7 写流程统一 cloud-first

耗材的 add / edit / delete 必须先要求登录 session 有效，先写云端再回本地。
云端失败：

- 用一条统一的「需联网」提示，不要在多处弹相似的对话框
- 本地状态保持原值，不做半成品落地

## 3. Tab / 页面索引这种东西不要硬编码

`tpCalibration = 6` 这种"默认布局下能猜对"的硬索引在多变 layout 下必爆。
正确做法：

- 用 `FindPage(<page_ptr>)` 拿到当前实际索引
- 重新插入时以"已知存在的兄弟页"作为 anchor，而不是绝对索引
- 任何 `RemovePage(<index>)` 之前都先按指针校验"这个 index 真的是我要删的页吗"

## 5. 自测 / 回归参考

每次再动耗材管理器，建议至少跑过这套快速回归：

| 场景 | 期望 |
| --- | --- |
| 全新装机 / multi_machine 关闭 → 启动 | Filament Manager Tab 在 Project / Calibration 之后正常出现，不丢 |
| macOS 反复点 Sidebar Tab：Project / Calibration / Filament Manager / Project | 切换流畅，无阻塞，无白屏 |
| Add Filament：拖 hue 后按 Esc | draft 丢弃，原色保留；OK 才提交 |
| 手动 spool：填 totalNetWeight=1500g、netWeight=200g | 云端 round-trip 后仍是 1500 / 200，不被压成 200/200 |
| 编辑耗材的非名字字段 | PUT 不返回 400 |
| Add/Edit Filament（macOS / Windows） | 品牌下拉条目一致，第三方品牌（HATCHBOX 等）都在；本地 spool 品牌也在 |
| Add Filament 类型下拉 | 名字归一（无 vendor 前缀）；zh_CN 下"耗材类型"标签翻译正常 |
| 离线 / 未登录尝试 add/edit/delete | 出现"需联网"提示一次，不写本地 |
| 材料列表 vs AMS 弹窗 | 同一台机器下，可见耗材集合一致 |

## 6. 后续改进建议

- 把 `build_preset_options()` 这类公共下拉构造函数统一拆成「切片用」和「UI metadata 用」两条入口，从签名上避免误用
- macOS WKWebView 调用统一封一层 async helper，禁掉同步 evaluateJavaScript 的直接调用点
- 颜色选择 / 类似 popover 模式抽公共组件（draft + commit + cancel/esc/click-outside 行为统一）
- 给 `spool_to_cloud_json` / 反向解析加一组 fixture 驱动的 round-trip 单测
- AMS 弹窗与 Filament Manager 列表的过滤逻辑抽公共函数，避免双份维护
