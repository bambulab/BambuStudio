## 1. 在 Studio 中的位置

耗材管理器解决的是“物理耗材库存”问题，它位于打印参数层和设备状态层之间，与二者保持关联但不直接替代它们。

| 层次 | 代表对象 | 作用 |
| --- | --- | --- |
| 打印参数层 | `PresetBundle::filaments` | 提供耗材预设、切片参数与 `setting_id` |
| 设备状态层 | `DevFilaSystem`、`DevAmsTray`、MQTT 推送 | 提供当前设备和 AMS 料槽状态 |
| 耗材库存层 | `FilamentSpool`、`wgtFilaManagerStore` | 维护“每一卷耗材”的本地库存台账 |
| 交互展示层 | `wgtFilaManagerPanel`、Web Panel 前端 | 提供列表、弹窗、筛选、AMS 同步等交互能力 |

这里补上的核心能力，是把“设备上当前看到的料”和“用户长期维护的本地库存”拆开建模，再通过同步策略把两者连接起来。

## 2. 当前整体架构

当前主线架构可以概括为 5 层：

1. 宿主层：`MainFrame` 和 `GUI_App` 负责 Tab 注册、生命周期、系统主题与设备更新入口。
2. 面板壳层：`wgtFilaManagerPanel` 负责 WebView 加载、消息路由和向前端推送数据。
3. 数据层：`wgtFilaManagerStore` 负责 `FilamentSpool` 的内存管理、CRUD 和本地 JSON 持久化。
4. 同步层：`wgtFilaManagerSync` 负责把设备 AMS 料槽状态映射回库存。
5. 前端层：耗材管理器 UI 负责列表、表单、筛选、分页、批量操作与主题表现。

当前推荐按“宿主 -> Panel -> Store/Sync -> Web 前端”的顺序理解整个系统。

### 当前主线实现形态

从 `20260416耗材管理器调整架构` 起，前端主线已经切换为标准 Web Panel 架构：

- 前端工程：`web-panels/`
- 面板源码：`web-panels/src/panels/fila-manager/`
- 构建产物：`resources/web/panels/fila-manager/`
- 通信协议：JSON-RPC 2.0，统一使用 `fila_manager.xxx`

旧版 `resources/web/fila_manager/` 的 HTML/CSS/JS 方案主要用于理解历史演进，不再作为本文件描述的长期主线架构。

## 3. 核心子系统职责

### 3.1 宿主与导航

- 在 `MainFrame` 中注册耗材管理器入口，并纳入顶级 Tab 体系
- 负责面板初始化、系统主题变化分发、宿主生命周期管理
- 不直接承载业务数据，而是通过 Panel 与 Store/Sync 协同

### 3.2 `wgtFilaManagerPanel`

- 加载耗材管理器前端页面
- 接收前端请求并分发到具体 handler
- 将最新的 spool 列表、主题变化等事件推送给前端
- 作为“宿主世界”和“前端世界”之间的边界层

在当前主线中，这一层的关键变化是协议已经从自定义 `{type, seq, command}` 迁移到 JSON-RPC 2.0。

### 3.3 `wgtFilaManagerStore`

- 维护 `spool_id -> FilamentSpool` 的核心数据模型
- 提供新增、更新、删除、收藏、存档、标空、批量操作等能力
- 负责本地 JSON 持久化与兼容读取
- 为前端列表展示和 AMS 同步提供统一数据源

这一层是耗材管理器的业务核心，决定“库存是什么”和“库存如何落盘”。

### 3.4 `wgtFilaManagerSync`

- 接收设备与 AMS 相关更新
- 按 `tag_uid` 优先、`setting_id + color` 次优先的策略匹配已有 spool
- 在无法匹配时按策略自动创建 spool
- 把设备侧状态收敛回库存模型，而不是直接让设备数据驱动页面

这一层的意义，是把“设备瞬时状态”转换成“库存持续状态”。

### 3.5 Web Panel 前端

- 展示耗材列表、空状态、搜索、筛选、排序、分页和批量操作
- 提供添加/编辑弹窗，支持手动录入与 AMS 读取双模式
- 负责主题切换、交互反馈和局部状态管理
- 通过 `bridge/client.ts` 与 C++ 层通信，不直接耦合宿主实现细节

这意味着前端主要负责交互编排，业务真源仍然在 C++ Store/Sync。

## 4. 关键数据流

### 4.1 页面初始化

1. 用户进入耗材管理器 Tab。
2. `wgtFilaManagerPanel` 加载前端页面。
3. 前端通过 `fila_manager.init` 或相关初始化请求拉起数据。
4. Panel 从 Store 读取库存、从其他入口补齐预设/机器/AMS 信息，并返回前端。
5. 前端渲染默认列表视图。

### 4.2 人工维护库存

1. 用户在前端执行新增、编辑、删除、收藏、存档等操作。
2. 前端通过 JSON-RPC 请求调用 `fila_manager.add_spool`、`fila_manager.update_spool` 等方法。
3. Panel 分发到 Store 执行业务更新并持久化。
4. 最新结果再通过响应或 `fila_manager.spool_list_updated` 推回前端。

### 4.3 AMS 同步

1. 设备或 MQTT 路径产生 AMS 料槽更新。
2. `GUI_App` 或相关宿主入口触发 `wgtFilaManagerSync`。
3. Sync 根据料槽信息匹配既有 spool；命中后用 percent × `effective_total_net_weight` 算出当前净重（克）写入 store。
4. **未匹配 → 静默跳过**，**不**自动新增 spool（STUDIO-18155 Q5）；新增料卷只走"添加耗材-从 AMS 读取" UI 入口。
5. **缺整卷净重 spool → 整条冻结**（STUDIO-18155 Q7），本地不刷 percent，云端不发 PUT；用户编辑补齐后下次 sync 自动恢复。
6. **identity 字段防御覆盖**：`Store::update_spool_if_changed` 强制保留既有 `tag_uid` / `color_code` / `setting_id` / `entry_method` / `cloud_synced`，sync 路径无法改写（STUDIO-18117 教训）。
7. 完成后联动云端 push（双轨节流：忙时 10 min cooldown + 差分；闲时直推）：
   - `compute_device_state(obj)` 一次 sync 算一次（is_in_printing / calibration / extrusion_cali / ams_status_main）
   - `wgtFilaManagerCloudSync::notify_ams_synced(changed, device_state)` 经 `AmsAutoPushThrottle` 决策
   - PUT body 仅含 `net_weight`，其余字段云端 PUT 不接受
   - 摘要通过 `auto_push_summary` ReportMsg 转发到前端 CloudBadge tooltip
8. Store 更新库存后，Panel 向前端推送最新列表。

**Sync 写权限**（STUDIO-18155 收口后）：

| 字段类别 | 字段 | sync 是否可写 |
|---|---|---|
| 余量 | `net_weight` / `remain_percent` / `status` | ✅ 可写 |
| 绑定 | `bound_dev_id` / `bound_ams_id` | ✅ 可写 |
| identity | `spool_id` / `tag_uid` / `color_code` / `setting_id` | ❌ 防御覆盖（store 强制保留既有值） |
| 元数据 | `brand` / `material_type` / `series` / `note` / `favorite` / `initial_weight` / `spool_weight` | ❌ store 直接忽略输入 |

### 4.4 主题切换

1. 宿主检测到系统主题变化。
2. Panel 向前端推送 `fila_manager.theme_changed`。
3. 前端切换 `dark` / `light` 主题表现。

## 5. 关键边界与设计口径

- 耗材管理器通过 `setting_id` 等字段关联 `PresetBundle`，但不改写预设存储结构。
- 耗材管理器读取设备与 AMS 状态，但不反向定义设备协议。
- 当前主线已以**云端为单一可信源**（cloud is source of truth）：本地 CRUD 走 push 串行队列上报；AMS 自动同步路径在 `notify_ams_synced` 内经 `AmsAutoPushThrottle` 决策后向云端 PUT 余量。
- AMS 自动同步对 store 的写权限**只覆盖余量 / 绑定 5 个字段**；identity 字段（tag_uid / color_code / setting_id 等）由 `Store::update_spool_if_changed` 防御覆盖，杜绝 STUDIO-18117 类污染。
- 前端可以做筛选、排序、分页等交互逻辑，但库存真值和持久化必须由 Store 保证。
- 架构演进允许更换前端技术栈，但不应破坏 `Store + Sync + Panel` 这条稳定职责分层。

## 6. 架构演进脉络

### 6.1 一期基线

`20260407耗材管理器一期框架` 建立了耗材管理器的基础分层：

- `Store`
- `Panel`
- `Sync`
- 旧版 WebView 前端

这一阶段完成了库存模型、AMS 同步、主页面和弹窗的整体闭环。

### 6.2 UI 与交互补强

`20260407耗材管理器UX对齐`、`20260408列表UI打磨`、`20260408品牌类型下拉联动`、`20260408AMS设备列表请求` 等变更，逐步把交互稿、设备请求链路和前端体验补齐。

### 6.3 Web Panel 架构迁移

`20260416耗材管理器调整架构` 将前端主线统一到标准 Web Panel 规范：

- 前端从原始 HTML/CSS/JS 迁移到 React + Zustand + Tailwind + TypeScript + Vite
- 通信从自定义协议迁移到 JSON-RPC 2.0
- 面板资源目录切换到 `resources/web/panels/fila-manager/`

因此，理解耗材管理器时需要区分“稳定业务分层”和“当前前端实现形态”两个层面：

- 稳定分层仍然是 `Host/Panel/Store/Sync/Frontend`
- 当前实现主线已经是 Web Panel，而不是旧版静态页面
